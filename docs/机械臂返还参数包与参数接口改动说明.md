# 机械臂返还参数包与参数接口改动说明

## 1. 需求背景

本次改动基于两份本地资料完成：

- `docs/[Talon] 六轴协作机械臂-通信协议_01.png`
- `docs/CRAIC-百度智能云智能服务机器人赛-机械臂调节教程.pdf`

目标不是在控制面直接驱动物理机械臂，而是在当前 `Embodied_Supervisor` 多仓结构里补齐两件事：

1. 把“机器人参数输入”收敛成可验证的共享契约。
2. 根据 Talon 协议把输入参数生成“机械臂返还参数包”，并提供后端获取接口给边端/机器人侧拉取。

## 2. 协议依据

### 2.1 通信协议图的直接依据

从 `docs/[Talon] 六轴协作机械臂-通信协议_01.png` 可以确认以下稳定约束：

- 帧头固定为 `0x42`。
- 地址 `Addr` 的 bit7 用于区分读写，`0x00~0x7F` 为写，`0x80~0xFF` 为读。
- `len = head + addr + len + data + check`，也就是总帧长。
- `check` 为前面各字节求和后取低 8 位。
- 典型控制地址为：
  - `0x09`：执行动作，整帧长度 `5`
  - `0x03`：单关节控制，整帧长度 `9`
  - `0x0A`：机械爪运动控制，整帧长度 `12`
  - `0x07`：关节使能，整帧长度 `6`
- 协议图备注里还给出“执行动作结束后发送结束信号”，因此本次在返回包里增加了 `expected_feedback` 字段。

### 2.2 调节教程 PDF 的直接依据

从 `docs/CRAIC-百度智能云智能服务机器人赛-机械臂调节教程.pdf` 第 6 章可以确认以下软件控制入口：

- `6.1 动作库执行`
  - 动作枚举稳定为：`reset / extend / contract / place / custom`
- `6.2 单关节控制`
  - 入口形态是 `setJointPosition(ArmJoint joint, float position)`
  - 关节枚举覆盖 `joint_1 ~ joint_6` 以及 `all`
  - 教程示例里明确出现 `joint_6 = 9.0`，用于机械爪张开距离
- `6.3 机械爪运动控制`
  - 入口形态是 `setClawMotion(float x, float y)`
  - 教程示例使用 `0.3` 一类浮点量，说明该接口本质上是双浮点参数控制

基于这两份资料，本次第一版实现只支持已经有稳定文档依据的 4 类命令：

- `execute_action`
- `set_joint_position`
- `set_claw_motion`
- `set_joint_enable`

没有把关节标定、动作库读写、心跳、系统复位直接做成开放接口，原因是当前项目更需要“参数包生成与获取闭环”，而不是一次性扩展成完整硬件控制台。

## 3. 设计结果

### 3.1 新增共享对象

新增两个中央契约对象：

- `RobotParameterInput`
  - 表示上层给机器人的参数输入
  - 核心字段包括 `command_kind`、`operator_id`、`reason`、`action_name`、`joint_name`、`position_value`、`claw_x`、`claw_y`、`enabled`
- `ArmCommandPacket`
  - 表示按 Talon USB/UART 协议生成的返还参数包
  - 核心字段包括 `packet_head`、`base_address`、`wire_address`、`frame_len`、`data_bytes`、`checksum`、`frame_bytes`、`frame_hex`、`target_topic`

### 3.2 参数包设计

返还参数包统一采用如下结构：

- `protocol = talon_usb_uart_v1`
- `packet_head = 0x42`
- `base_address = Talon 功能地址`
- `wire_address = 当前写命令直接等于 base_address`
- `frame_len = 4 + data_bytes.length`
- `checksum = sum(frame_bytes[:-1]) & 0xFF`
- `target_topic = embodied/{siteId}/{robotId}/command`
- `input = 原始 RobotParameterInput`

当前 4 类命令的映射如下：

| command_kind | base_address | 数据区 |
| --- | --- | --- |
| `execute_action` | `0x09` | `action_code(1 byte)` |
| `set_joint_position` | `0x03` | `joint_code(1 byte) + float32(position)` |
| `set_claw_motion` | `0x0A` | `float32(x) + float32(y)` |
| `set_joint_enable` | `0x07` | `joint_code(1 byte) + enable_flag(1 byte)` |

动作编码采用教程里的稳定顺序：

- `reset -> 0x01`
- `extend -> 0x02`
- `contract -> 0x03`
- `place -> 0x04`
- `custom -> 0x05`

## 5. 当前边界

这次实现明确只覆盖“参数包生成与拉取”：

- 已实现：
  - 输入参数校验
  - Talon 协议包生成
  - 本地持久化
  - latest 查询接口
  - 边端 checksum 校验辅助
- 未实现：
  - 物理串口发送
  - 真机 ACK/结束信号监听
  - 标定类开放接口
  - 动作库备份/导入导出接口

这样做的原因是当前仓库还是教师侧监督系统，不应该在第一版里直接膨胀成完整机械臂调试软件。

## 6. 验证结果

### 6.1 单元测试

通过：

- `data-access-layer`: `3 passed`
- `backend-api-service`: `3 passed`
- `edge-client-sdk`: `15 passed`

### 6.2 Live Smoke

已实际启动本地后端并完成以下链路：

1. 创建 session
2. 提交 `execute_action/reset` 类型的 `RobotParameterInput`
3. 成功返回参数包
4. 再次通过 latest 接口取回同一参数包

现场结果：

- `frame_hex = 42 09 05 01 51`
- `target_topic = embodied/lab-a/dual-arm-station-01/command`

说明“输入参数 -> 返还参数包 -> latest 获取”链路已经真实可用。
