Embodied Supervisor / 督教智能

教师侧监督系统模块架构蓝图

这份设计基于 reference/初选项目设计方案书.pdf 的业务闭环、reference/模块化与敏捷并行... 的多仓库原则， 以及当前 yolodemo 已经验证的三个原型能力：SOP / todo 驱动的视频节点切分、教师侧实时监督台雏形、离线网页回放包导出。 设计目标不是做“通用机器人控制平台”，而是做一套独立于学生侧执行闭环之外的教师侧监督、纠偏和教学资产化系统。

业务定位 教师侧监督智能体，不直接接管底层运动控制 核心闭环 感知 → 判断 → 介入 → 沉淀 部署原则 边缘实时闭环优先，云端增强推理按需介入 工程组织 5 仓库 + 契约优先 + 结构化 JSON 事件流
边缘优先

把高频监督留在 DK-2500

多路视频接入、时间轴对齐、轻量检测、SOP 状态机和阈值告警都先在边缘侧完成，避免云端往返时延破坏实时纠偏。

契约优先

先定义对象，再让模块互联

所有跨模块对象统一沉淀到 shared-models-and-apis：TaskContextFrame、DeviationEvent、GuidanceSuggestion、TeachingCard。

云端增强

复杂异常再升级到 VLM

边缘侧只负责“看得快、判得稳”，云端只在低置信度、复杂因果和需要自然语言解释时介入，避免把整个系统做成“云端全知全能”。

资产闭环

失败样本不止告警，还要沉淀

关键帧、短视频片段、偏差类型、指导文本和纠偏前后对比统一沉淀为教学卡片与训练候选样本，支撑后续 RAG、复盘和模型微调。

总体架构图

总体结构按“外部数据源、边缘监督、云端控制面、数据层、教师工作站、中央契约层”组织。 其中 yolodemo 的当前能力主要落在边缘监督和回放资产这两块，未来应被拆解并迁入正式仓库，而不是继续作为一个单仓脚本项目堆功能。

终端输入层 → capture-gateway

现场数据接入。视频走 RTSP/UVC，音频走 I2S/USB，机器人状态建议走 gRPC RobotState.Subscribe()。

capture-gateway → timeline-context

内部对象流：FrameEnvelope、AudioEnvelope、RobotStateFrame。目标是统一时间基准，而不是让监督引擎自己拉流。

timeline-context → sop-supervisor

输入对象：TaskContextFrame。监督引擎基于该对象输出 TaskNodeProgress、DeviationEvent、InterventionTrigger。

sop-supervisor → uplink-agent

边缘先做初判，复杂异常再升级。上传载荷应包含节点 ID、偏差类型、置信度、关键帧引用和摘要指标。

uplink-agent → supervision-ingest-api

边缘入云三条主 API：POST /api/v1/sessions、POST /api/v1/sessions/{id}/edge-events:batch、POST /api/v1/sessions/{id}/media。

supervision-ingest-api → data-access-layer

写入会话、事件和媒体索引：SessionRepository.upsert()、EventRepository.batchInsert()、MediaObjectRepository.finalize_object()。

guidance-orchestrator → 云端 VLM / RAG

复杂异常建议接口：POST /api/v1/guidance/advise。请求体应包含关键帧、节点上下文、历史相似案例、Prompt 模板版本。

guidance-orchestrator / asset-service → data-access-layer

RAG 查询走 EmbeddingRepository.search()；教学卡片与导出任务写入 PostgreSQL，媒体落对象存储，向量写入 Qdrant。

realtime-session-hub → teacher-workstation

前台实时消费：GET /api/v1/sessions/{id} + GET /api/v1/sessions/{id}/stream。v1 继续沿用 SSE，避免过早引入 WebSocket 复杂性。

teacher-workstation → realtime-session-hub → uplink-agent

人工介入从工作站发起：POST /api/v1/sessions/{id}/interventions，后端再通过 MQTT embodied/{site}/{robot}/guidance 或 /command 下发。

两条教学闭环

内闭环：单次任务实时监督

  1. 边缘侧接入多路视频、音频和机器人状态，统一到同一时间轴。
  2. 使用轻量检测 + SOP 状态机完成时序、空间、逻辑三类偏差筛查。
  3. 低复杂度异常直接在边缘生成告警；复杂异常打包关键帧和上下文升级到云端。
  4. 后端通过 realtime-session-hub 以 SSE 推送状态流，工作站显示节点进度、关键事件和介入建议。
  5. 教师确认后可下发指导文本或复位命令，但不直接接管学生侧轨迹控制。

外闭环：失败样本资产化

  1. 边缘或后端截取关键帧、异常片段、上下文窗口与度量指标。
  2. 后端将元数据写入 PostgreSQL，将视频与图片落到 MinIO/S3，将 embedding 写入 Qdrant。
  3. RAG 检索历史相似案例，生成结构化教学卡片。
  4. 工作站支持回放、检索、对比和导出，用于复盘、答辩展示与后续训练数据集构建。

核心 API 面与事件契约

控制面 OpenAPI

POST   /api/v1/sessions
GET    /api/v1/sessions/{sessionId}
POST   /api/v1/sessions/{sessionId}/edge-events:batch
POST   /api/v1/sessions/{sessionId}/media
POST   /api/v1/sessions/{sessionId}/guidance-requests
GET    /api/v1/sessions/{sessionId}/stream
POST   /api/v1/sessions/{sessionId}/interventions
  • sessions 用来标识一次任务执行或一次演示过程。
  • edge-events:batch 用来接收边缘侧批量上送的节点进度、偏差与心跳。
  • stream 是教师工作站消费的状态流接口，v1 使用 SSE。

资产与检索 API

POST   /api/v1/assets/teaching-cards:generate
GET    /api/v1/assets/teaching-cards
POST   /api/v1/search/semantic
POST   /api/v1/exports/dataset
GET    /api/v1/assets/{assetId}
  • 教学卡片生成不应只依赖 LLM 文本，还必须关联关键帧、片段和节点上下文。
  • 语义检索接口同时支持文本 query、标签 filter 和时间范围过滤。

MQTT Topic 建议

embodied/{siteId}/{robotId}/heartbeat
embodied/{siteId}/{robotId}/events
embodied/{siteId}/{robotId}/alerts
embodied/{siteId}/{robotId}/guidance
embodied/{siteId}/{robotId}/command
  • events 发布结构化 JSON 事件，适合低延迟、弱网络条件下的边缘上送。
  • command 只用于指导文本、软暂停、复位等教师侧控制，不扩展为底层运动轨迹通道。

协议矩阵

源 目标 主协议 备选 原因
摄像头 / 麦克风 capture-gateway RTSP / UVC / USB / I2S 本地文件回放 现场接入必须兼容工业和服务机器人常见外设，调试期支持离线视频回放复现实验。
学生侧机器人控制器 capture-gateway gRPC streaming MQTT / Modbus-TCP 状态流需要强类型和低时延；具体协议可按机器人平台兼容，但语义对象必须统一。
边缘内部模块 边缘内部模块 进程内 asyncio.Queue + Pydantic DTO 本地 JSONL 落盘 方案书强调结构化 JSON 事件流；本地同时保留 JSONL 有利于复盘与断网补传。
uplink-agent supervision-ingest-api MQTT 5.0/TLS（事件） + HTTPS（媒体） HTTP batch 事件与媒体分流：事件讲时效和轻量，媒体讲可靠和大对象上传。
guidance-orchestrator 云端 VLM HTTPS JSON 厂商 SDK 多模态模型供应商差异较大，统一成 JSON adapter 最利于后续切换云端服务。
realtime-session-hub 教师工作站 SSE WebSocket 当前原型已使用 EventSource 跑通一对多状态流，v1 先用简单稳定的一向流模式。
asset-service 媒体 / 向量 / 元数据存储 S3 API + PostgreSQL + Qdrant 本地文件系统 不同数据形态分仓存储：关系数据、媒体对象、语义向量各走最适合的底座。
教师工作站 机器人复位接口 REST → MQTT / gRPC 桥 人工线下复位 教师系统只做审计可追踪的指导与复位，不扩展成高风险的实时运动控制面。
