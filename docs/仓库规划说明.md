# 仓库规划说明

## 设计原则

- 按技术边界拆仓，不按“谁来做”拆仓。
- 共享契约优先，避免前后端和客户端各自维护一份数据结构。
- 每个仓库独立发布、独立测试、独立打标签。
- `main` 只保留稳定代码，实验性内容走短分支。

## 五仓库职责

### 1. `edge-client-sdk`

适合放：

- 设备接入逻辑
- 本地缓存、离线同步
- 与后端通信的底层封装
- 运行在边缘端的通用工具

不适合放：

- 页面 UI
- 业务后台接口实现
- 数据库访问代码

### 2. `frontend-web-app`

适合放：

- Web 页面
- 组件、路由、状态管理
- 调用后端 API 的前端业务逻辑

不适合放：

- 数据库直连逻辑
- 共享契约的源定义

### 3. `backend-api-service`

适合放：

- 对外 API
- 核心业务规则
- 权限校验
- 服务编排

不适合放：

- 直接耦合页面逻辑
- 到处散落的数据表访问代码

### 4. `data-access-layer`

适合放：

- 数据库连接与查询封装
- 仓储层
- 缓存封装
- 数据持久化策略

不适合放：

- 控制器
- 页面组件
- 跨层业务流程

### 5. `shared-models-and-apis`

适合放：

- OpenAPI 文档
- JSON Schema
- DTO 定义
- 示例请求和响应
- 跨仓库共享常量

这是整个项目最重要的“契约仓库”。接口变更先改这里，再推动上下游仓库适配。

## 分支建议

- 稳定分支：`main`
- 功能分支：`feature/<topic>`
- 实验分支：`experiment/<topic>`
- 修复分支：`fix/<topic>`

## 版本建议

- 所有仓库都用语义化版本：`MAJOR.MINOR.PATCH`
- `shared-models-and-apis` 发生不兼容变更时，下游仓库同步评估大版本升级
- 每完成一个阶段，就在 `main` 上打标签，例如 `v0.1.0`、`v0.2.0-beta.1`
